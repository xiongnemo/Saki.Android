# Internationalization Guide

This document is a draft for Saki.Android internationalization work. It has two audiences:

- Contributors who want to add or improve translations.
- AI agents that add, use, and maintain user-facing strings in the app.

The app uses Android resource strings as the source of truth for localizable UI text.

## For Contributors

### Goal

Translations should make the app feel natural in the target language while preserving the meaning, tone, and functional clarity of the default English strings.

The default language lives in:

```text
app/src/main/res/values/strings.xml
```

Translated strings should live in Android locale-specific resource folders, for example:

```text
app/src/main/res/values-zh/strings.xml
app/src/main/res/values-ja/strings.xml
app/src/main/res/values-fr/strings.xml
```

If this work later moves to a separate i18n repository, contributors should submit translation files there using the same resource names and XML structure.

### What To Translate

Translate user-facing text only:

- Screen titles, section labels, tabs, buttons, menu items, empty states, and dialogs.
- Snackbar messages and form validation errors.
- Accessibility labels when present.
- Pluralized messages under `<plurals>`.

Do not translate:

- Resource names such as `settings_title` or `message_backup_exported`.
- Format placeholders such as `%1$s`, `%2$d`, or `%1$.1f`.
- Technical protocol names such as Subsonic, Bluetooth, HTTP, URL, API, or app/product names unless the target language normally localizes them.
- Values that come from a server, song title, artist name, album title, filename, or user input.

### File Rules

Each translation file must be valid Android XML:

```xml
<resources>
    <string name="settings_title">Settings</string>
</resources>
```

Keep the same keys as the default file. A translated file may start incomplete, but every translated key must match an existing default key.

For strings with placeholders, keep the same placeholders and types:

```xml
<!-- Default -->
<string name="message_added_to_queue">Added %1$s to the queue.</string>

<!-- Translation: keep %1$s -->
<string name="message_added_to_queue">...</string>
```

For plural resources, keep all required quantities for the target language. Android commonly requires at least `one` and `other`, but some languages need additional categories.

```xml
<plurals name="message_cleared_download_count">
    <item quantity="one">%1$d download cleared.</item>
    <item quantity="other">%1$d downloads cleared.</item>
</plurals>
```

### Translation Style

Prefer natural UI language over literal word-by-word translation.

Keep text concise because many strings appear inside compact mobile controls. Avoid adding extra explanation unless the English source is unclear or the target language requires it.

Use consistent translations for repeated UI concepts:

- Server
- Endpoint
- Queue
- Download
- Stream cache
- Cover art
- Playlist
- Now playing
- Offline playback

When a string contains an action, preserve the action:

- "Save" should remain an imperative action.
- "Saved" should remain a completed state.
- "Unable to..." should remain an error state.

### Submitting A Translation

A translation contribution should include:

- The locale being added or updated, for example `zh`, `zh-rTW`, `ja`, or `fr`.
- The changed `strings.xml` file.
- A short note about whether the translation is complete or partial.
- Screenshots when the change affects layout-sensitive strings, especially navigation, buttons, and settings rows.

Before submitting, check:

- The XML parses.
- Placeholders are unchanged.
- Plural entries keep valid quantity names.
- No English text was accidentally removed from the default `values/strings.xml`.
- Long translations do not obviously overflow common mobile UI elements.

Suggested local checks:

```powershell
[xml](Get-Content -Raw app\src\main\res\values-zh\strings.xml) | Out-Null
.\gradlew.bat lintDebug
.\gradlew.bat assembleDebug
```

The Gradle commands require a configured Java environment.

### If A Separate i18n Repository Is Created

The separate repository should preferably contain:

- One folder per Android locale.
- A copy of the default `values/strings.xml` for reference.
- A status table showing completion by locale.
- Contributor notes for tone, terminology, and known untranslatable terms.

The Android app repository should still treat resource names as stable API. Renaming keys should be rare because it creates unnecessary churn for translators.

## For AI Agents

### Prime Directive

Do not introduce new hardcoded user-facing English strings in Kotlin UI or ViewModel code.

Use Android string resources for all stable user-facing text. Dynamic server messages, filenames, song titles, artist names, and user input may remain dynamic values.

### Where Strings Belong

Default strings belong in:

```text
app/src/main/res/values/strings.xml
```

Compose UI should read resources with:

```kotlin
stringResource(R.string.some_key)
pluralStringResource(R.plurals.some_count, count, count)
```

Non-Compose layers such as ViewModels should use `UiText` instead of resolving Android resources directly:

```kotlin
UiText.resource(R.string.error_start_playback)
UiText.resource(R.string.message_added_to_queue, song.title)
UiText.plural(R.plurals.message_cleared_download_count, removed, removed)
throwable.localizedOr(R.string.error_start_playback)
```

Resolve `UiText` at the UI boundary:

```kotlin
snackbarHostState.showSnackbar(message.asString(context))
Text(text = error.asString())
```

### Naming Conventions

Use stable, descriptive resource names grouped by feature or purpose.

Recommended prefixes:

- `common_` for reusable UI words such as OK, Cancel, On, Off.
- `settings_` for settings screen labels and descriptions.
- `server_config_` for server setup and endpoint editing.
- `browse_` for library browsing screens.
- `player_` for playback and now-playing UI.
- `message_` for snackbars and success/info messages.
- `error_` for user-facing errors and fallback errors.

Prefer names based on meaning, not exact English copy:

```xml
<string name="error_start_playback">Unable to start playback.</string>
```

Avoid names that encode punctuation or temporary wording:

```xml
<!-- Avoid -->
<string name="unable_to_start_playback_period">Unable to start playback.</string>
```

### Adding A New String

When adding a new user-facing string:

1. Add a default English entry to `app/src/main/res/values/strings.xml`.
2. Use the resource from Kotlin instead of hardcoding text.
3. If the text contains a number-dependent noun, use `<plurals>`.
4. If the text contains dynamic values, use positional placeholders such as `%1$s` or `%1$d`.
5. Run XML parsing and a whitespace diff check.

Example:

```xml
<string name="message_added_to_queue">Added %1$s to the queue.</string>
```

```kotlin
snackbarMessages.emit(UiText.resource(R.string.message_added_to_queue, song.title))
```

### Placeholders

Always use indexed placeholders:

```xml
<string name="message_saved_offline">Saved %1$s for offline playback at %2$s.</string>
```

Indexed placeholders let translators reorder values naturally.

Keep placeholder types correct:

- `%1$s` for strings.
- `%1$d` for integers.
- `%1$.1f` for formatted decimals.

If a placeholder value is itself localizable, pass it as `UiText` or use a resource ID at the UI layer:

```kotlin
UiText.resource(
    R.string.message_stream_quality_set,
    UiText.resource(quality.labelRes()),
)
```

### Plurals

Use `<plurals>` for quantity-sensitive messages. Do not build English plurals manually in Kotlin.

```xml
<plurals name="message_cleared_stream_cache_entry_count">
    <item quantity="one">Cleared %1$d stream cache entry.</item>
    <item quantity="other">Cleared %1$d stream cache entries.</item>
</plurals>
```

```kotlin
UiText.plural(R.plurals.message_cleared_stream_cache_entry_count, removed, removed)
```

The first numeric argument after the resource ID is the quantity Android uses to choose the plural form. Pass the display count again when the string includes `%1$d`.

### Dynamic And Remote Text

Some text should remain dynamic:

- Subsonic server error messages.
- HTTP status details.
- Song, album, artist, playlist, and server names.
- File names, URLs, and user-entered values.

If a dynamic error exists, preserve it with `UiText.dynamic(...)`. If it is missing or blank, use a localized fallback:

```kotlin
throwable.localizedOr(R.string.error_load_albums)
```

Avoid turning arbitrary exception messages into default English strings. They may contain technical details, server text, or values that should not become translation keys.

### Enum And Option Labels

Enums and settings options must expose string resources, not hardcoded labels.

Use helper mappings such as:

```kotlin
quality.labelRes()
mode.labelRes()
strategy.labelRes()
textScale.labelRes()
```

Do not use enum `.name` or custom `.label` strings for UI text unless the label is already backed by a resource.

### Compose Guidelines

Inside composables:

- Use `stringResource` for simple strings.
- Use `pluralStringResource` for plural strings.
- Use `UiText.asString()` only when receiving a `UiText` from state or events.
- Keep leaf components simple when possible. It is acceptable for a screen to resolve `UiText` before passing `String?` to small display-only components.

Avoid calling Android `Context.getString(...)` directly from composables when `stringResource(...)` works.

### ViewModel Guidelines

ViewModels should not depend on `Context` just to format UI text.

Use `UiText` in UI state and one-shot UI events:

```kotlin
private val snackbarMessages = MutableSharedFlow<UiText>(extraBufferCapacity = 1)

data class ScreenState(
    val error: UiText? = null,
)
```

This keeps business logic testable and lets the UI resolve the correct language at display time.

### Maintenance Checks

Before finishing an i18n-related change, run the checks that are available in the environment:

```powershell
[xml](Get-Content -Raw app\src\main\res\values\strings.xml) | Out-Null
git diff --check
.\gradlew.bat lintDebug
.\gradlew.bat assembleDebug
```

If Gradle cannot run because Java is unavailable, state that clearly in the handoff.

Useful searches:

```powershell
Select-String -Path app\src\main\java\com\anzupop\saki\android\**\*.kt -Pattern 'Text\("|\bButton\(.*"|\bsnackbarMessages\.(emit|tryEmit)\("|throwable.message \?: "'
```

Manual search still matters. Some hardcoded strings are split across composables, previews, and test data.

### Previews And Tests

Preview-only strings may be hardcoded when they are clearly fake content and not shipped to users. Prefer resource-backed text when the preview exercises real UI labels.

Tests may use hardcoded strings for fixtures, but tests that assert UI labels should prefer resource IDs or shared constants when practical.

### Changing Existing Strings

Changing the English value of a string is allowed when improving clarity, but be aware that it may invalidate existing translations.

Renaming a resource key should be avoided unless:

- The old name is misleading.
- The string moved to a different feature area.
- The key conflicts with a clearer naming pattern.

When renaming, update all locale files in the same change or document that translations need follow-up.

### Do Not Do This

Do not concatenate localizable sentences in Kotlin:

```kotlin
"Imported " + count + " server(s)"
```

Do not manually pluralize English:

```kotlin
"Cleared $count download${if (count == 1) "" else "s"}."
```

Do not use raw enum labels in UI:

```kotlin
quality.label
```

Do not resolve resources in ViewModels by passing around Android `Context` solely for strings.

Do not add a translation key for a value that comes from the user's media library or server.
